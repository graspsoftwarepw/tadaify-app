# Logo motion v8 — Orbit the Stage

## User's verbatim brief (Polish)

> "Powinno być tak, że on idzie w prawo i robi dookoła brzegów całe 360 stopni, a dopiero potem wraca do środka. Będąc na brzegach, po prostu tak delikatnie, bardzo subtelnie się buja."

**English translation:**
> "It should be like this: it goes to the right and makes a full 360 degrees around the edges, and only then returns to the centre. While at the edges, it just gently, very subtly sways."

## Iteration history

| Version | Motion | Rejection reason |
|---------|--------|-----------------|
| v4 | Waving arms keyframes | Too chaotic — too many moving parts |
| v5 | Gentle L↔R sway, slow period | "Za wolno" — too slow, not enough presence |
| v6 | Step right, hold, return, alternate sides | "Galareta" — too static, too much rest time |
| v7 | Continuous L↔R sway, same direction | Still continuous but still pendulum-like — no orbit |
| **v8** | **Full 360° clockwise orbit + radial sway** | — |

## Orbit geometry

### Dimensions (SVG 100×100 viewBox)

| Element | Value | Notes |
|---------|-------|-------|
| Large orb (stage) radius R | 40 SVG units | cx=50 cy=50 |
| Warm orb (creator) radius | 20 SVG units | = R/2 |
| Orbit radius (warm orb center path) | 18.4 SVG units | = R/2 × 0.92 (8% inset margin) |
| Warm orb outer reach at orbit edge | 38.4 SVG units | 18.4 + 20 = 38.4 < 40 → stays inside |

### Why 8% inset?

At exactly R/2 = 20 SVG units orbit radius, the warm orb outer edge would sit precisely on the large orb perimeter — pixel-perfect but visually harsh. An 8% inset (orbit radius = 18.4) gives a subtle visual margin so the warm orb reads as "comfortably inside" the stage, not clipping it.

### Keyframe positions

| Clock position | Angle | cx | cy |
|---------------|-------|-----|-----|
| 3 o'clock (start) | 0° | 68.4 | 50.0 |
| 6 o'clock | 90° | 50.0 | 68.4 |
| 9 o'clock | 180° | 31.6 | 50.0 |
| 12 o'clock | 270° | 50.0 | 31.6 |
| 3 o'clock (end) | 360° | 68.4 | 50.0 |

Direction: **clockwise** (in SVG screen coordinates, angle increases → y increases first = downward first → matches reading "right → down → left → up").

## Sway option selected: Option A — Radial breathing

**Rationale:** The user's description "delikatnie, bardzo subtelnie się buja" (gently, very subtly sways) maps best to a radial breath — the orb quietly pulses in and out from the stage center while walking the circle. It reads as "alive and present" without adding a second axis of motion or any oscillation that could be confused with the v5/v7 lateral sway.

Option B (angular velocity variation) would make the orbit appear to stutter or rush — not "subtle sway". Option C (both) risks over-engineering a gesture that should be nearly invisible.

### Sway parameters

| Parameter | Value |
|-----------|-------|
| Amplitude | ±0.55 SVG units (±3% of orbit radius 18.4) |
| Frequency | 4 cycles per full 360° orbit |
| Period (one sway cycle) | 1.4 s |
| Frequency in Hz | ≈ 0.71 Hz |

At 240px rendered size, ±0.55 SVG units = ±1.3 px of radial movement. Barely perceptible — exactly "very subtle".

## Full timing table

| Phase | Start | Duration | Easing | Description |
|-------|-------|----------|--------|-------------|
| 1 — Center → 3oc | 0.0 s | 0.9 s | ease-out cubic | Horizontal glide rightward |
| 2 — 360° orbit | 0.9 s | 5.6 s | Linear + sway | Full clockwise circuit with radial breathing |
| 3 — 3oc → Center | 6.5 s | 0.9 s | ease-in cubic | Horizontal glide back |
| 4 — Rest | 7.4 s | 0.8 s | — | Static brand mark |
| **Total cycle** | — | **8.2 s** | — | Loop |

Motion is "active" (travelling) for 7.4 s out of 8.2 s = **90% of the cycle**. Only 10% is rest. This directly addresses the v5 "too slow" and v6 "too static" feedback.

## Accessibility

`@media (prefers-reduced-motion: reduce)` kills the JS animation entirely. The warm orb remains at (50,50) — the static centered brand mark. Favicon sizes (32px, 16px) are always static regardless of motion preference.

## Implementation approach: JavaScript RAF engine

**Chosen over CSS `offset-path`** for the following reasons:

1. **Radial sway is trivial in JS** — just add `SWAY_AMP * sin(SWAY_FREQ * angle)` to the radius. In pure CSS, layering a radial breathing animation on top of an `offset-path` orbit requires a wrapper element + nested keyframes + careful timing synchronisation.
2. **Phase control is precise** — easing per phase (ease-out glide, linear orbit, ease-in return) is cleanly expressed as three `if` branches. CSS would require percentage arithmetic across a single `animation` timeline.
3. **Single shared tick** — all logo sizes (hero 240px, nav 48px, nav lockup) share one `requestAnimationFrame` loop → they stay in sync automatically.

### Core snippet

```js
const ORBIT_R   = 18.4;  // SVG units
const CX = 50, CY = 50;  // stage center
const SWAY_AMP  = 0.55;  // ±3% of orbit radius
const SWAY_FREQ = 4;     // cycles per full orbit

function getPos(t) {
  // t: ms within current cycle [0, 8200)
  if (t < 900) {
    // Phase 1: ease-out glide to 3oc
    const p = 1 - Math.pow(1 - t/900, 3);
    return { x: CX + ORBIT_R * p, y: CY };
  } else if (t < 6500) {
    // Phase 2: clockwise orbit + radial sway
    const frac  = (t - 900) / 5600;
    const angle = 2 * Math.PI * frac;
    const r     = ORBIT_R + SWAY_AMP * Math.sin(SWAY_FREQ * angle);
    return { x: CX + r * Math.cos(angle), y: CY + r * Math.sin(angle) };
  } else if (t < 7400) {
    // Phase 3: ease-in return
    const p = Math.pow((t - 6500) / 900, 3);
    return { x: CX + ORBIT_R * (1 - p), y: CY };
  } else {
    return { x: CX, y: CY }; // Phase 4: rest
  }
}
```

### CSS offset-path alternative (not used, documented for reference)

```css
@keyframes orbit360 {
  from { offset-distance: 0%; }
  to   { offset-distance: 100%; }
}
.orb-warm {
  offset-path: path("M 68.4 50 A 18.4 18.4 0 1 1 68.399 50.001 Z");
  animation: orbit360 5.6s linear infinite;
}
```

Limitation: adding radial sway on top requires a second `transform: scale()` keyframe animation, and the two animations' phases must be manually synced. JS is cleaner here.
