# Logo Motion v4 — Wave-at-Edges

## The metaphor (user's words, verbatim)

> Yellow circle = the creator, the person who owns the page.
> Purple circle = the stage.
> Motion should feel like: the creator walks to the edge of the stage to greet, waves to the audience by rocking side to side (buja się lewo-prawo), returns to center, goes to the other edge, waves again, returns. And so on.

---

## Full spec

### Sizing (locked, unchanged from v3)

| Element | Value |
|---|---|
| viewBox | `0 0 100 100` |
| Large orb (stage) | `cx=50 cy=50 r=40` — fills 80% of canvas |
| Warm orb (creator) | `r=20` — exactly ½ of large orb radius |
| Static position | Warm orb centered on large orb: `cx=50 cy=50` |
| Travel distance | Large orb r − warm orb r = 40 − 20 = **20 SVG units** |
| Wave amplitude | **±10 SVG units** perpendicular to radius (12.5% of orb diameter) |

When the warm orb travels to an edge, its outer edge aligns with the large orb's perimeter (warm orb center sits at 20 units from the large orb's center). The warm orb does **not** spill outside the stage circle at any point — including during the wave — because the perpendicular wave displacement (±10 units) stays well within the circle boundary.

*Edge geometry check: at North edge, center = (50, 30), r=20. Any point of the warm orb is at most 20 units from (50,30). The farthest point from the large orb center (50,50) is 20+20=40 — exactly flush with the large orb's perimeter. The wave moves tangentially (East-West at North position), so the orb stays on the surface of the stage, not outside it.*

### Wave gesture direction per edge

| Edge | Orb center at edge | Wave axis | Wave reads as |
|---|---|---|---|
| North (12 o'clock) | `(50, 30)` | X (East ↔ West) | Hand waving left-right |
| East (3 o'clock) | `(70, 50)` | Y (North ↔ South) | Hand waving up-down |
| South (6 o'clock) | `(50, 70)` | X (East ↔ West) | Hand waving left-right |
| West (9 o'clock) | `(30, 50)` | Y (North ↔ South) | Hand waving up-down |

The greeter sequence is **clockwise: N → E → S → W**. This is the most natural reading direction and matches how a performer would acknowledge all four sides of a theatre.

### Timing table

| Phase | Duration | Easing | Notes |
|---|---|---|---|
| Long center rest | 1 800ms | — | Static lock state. Brand mark at rest. Loop anchor point. |
| Travel to edge | 580ms | `ease-out` | Smooth purposeful walk. No overshoot — the creator is walking, not bouncing. |
| **Wave at edge (3 cycles)** | **900ms** | `cubic-bezier(0.45,0,0.55,1)` | **Star of the animation.** 7 keyframes producing 3 full left-right-left rocking cycles. Each half-cycle ≈ 150ms. |
| Return to center | 480ms | `ease-in-out` | Slightly faster than outbound (walking back, not strolling). |
| Center pause | 280ms | — | Brief gathering moment. Enough to "land" before the next greeting. |
| **Full greeting (one direction)** | **2 240ms** | — | 580 + 900 + 480 + 280 |
| **Full cycle (4 greetings + rest)** | **≈ 10 760ms** | — | 4 × 2 240ms + 1 800ms ≈ **10.8s** |

### Keyframe structure for wave gesture

The wave uses 7 keyframes at equal intervals (0%, 16.7%, 33.3%, 50%, 66.7%, 83.3%, 100%):

```
t=0%:    edge center (just arrived, no displacement)
t=16.7%: +amplitude in wave direction  ← first peak
t=33.3%: edge center                   ← crossing
t=50%:   -amplitude                    ← second peak
t=66.7%: edge center                   ← crossing
t=83.3%: +amplitude                    ← third peak
t=100%:  edge center                   ← settle (ready to return)
```

Produces exactly 3 full oscillation cycles. The easing (`cubic-bezier(0.45,0,0.55,1)`) is approximately sinusoidal — gentle deceleration approaching each peak, smooth through the center crossing.

## Why v3 failed

v3's warm orb arrived at the edge with a **spring overshoot**: it traveled to 26 SVG units past center (clipping slightly outside the large orb), then settled back to 20 units. The intent was "pop to say hello."

**The problem:** the overshoot pattern (travel past target → snap back) is the universal CSS/physics "spring" idiom. Viewers read it as a bounce, a load indicator, or elastic physics — not as a greeting gesture. There is no "waving hello" signal in an overshoot.

Additionally, the v3 peek pause was a static hold — the orb arrived at the edge and just... stopped there. A person standing at the edge of a stage and freezing does not read as greeting. It reads as a loading state.

---

## Why v4 works

v4 separates **travel** from **gesture** into two distinct phases:

1. The orb walks to the edge (travel phase — unremarkable, just getting there).
2. The orb *does something* at the edge: it rocks side-to-side 3 times (wave phase — the actual greeting).

The rocking motion maps directly to the "machanie ręką" (hand-waving) gesture. The motion is perpendicular to the direction of travel, which is the geometric equivalent of a person standing still and waving their hand. There is no "bouncing" component — the center of the orb stays locked at the edge position in the radial direction throughout the wave.

Three cycles is the minimum number to read unambiguously as "waving" rather than "wobbling" — one cycle would read as a glitch, two as uncertain. Three is the Polish `cześć` wave: once isn't a greeting, three times is.

## Implementation options for production

### Option A: Web Animations API (current implementation)

**Used in this mockup.** Async function chain via `Promise`-based `element.animate()`. Each phase is a separate animation object that resolves on `onfinish`.

```js
async function doGreeting(el, dir) {
  await travel(el, dir.edge);    // animate to edge
  await wave(el, dir);           // 7-keyframe wave
  await returnCenter(el);        // animate back
}
```

**Pros:** precise per-phase control, no CSS collision, works in all modern browsers, easy to adjust timing per phase.
**Cons:** JS-dependent (degrades gracefully to static if JS disabled), not declarative.

### Option B: CSS `@keyframes` with `animation-composition: add`

Would allow two simultaneous animations: one for radial position, one for perpendicular wave. Requires `animation-composition: add` (supported in Chrome 112+, Firefox 115+, Safari 16+).

**Pros:** purely declarative.
**Cons:** `animation-composition: add` has incomplete support. Debugging two independently-timed keyframe tracks that must stay synchronized is fragile. The wave timing (3 cycles in 900ms) interacts badly with the outer animation's percentage-based keyframes.

### Option C: Framer Motion (React production)

```jsx
const waveVariants = {
  initial: { x: edgeX, y: edgeY },
  waving: {
    x: [edgeX, edgeX + amp, edgeX, edgeX - amp, edgeX, edgeX + amp, edgeX],
    y: edgeY,
    transition: { duration: 0.9, ease: 'easeInOut', times: [0, 0.167, 0.333, 0.5, 0.667, 0.833, 1] }
  }
}
```

Uses Framer Motion's `useAnimate()` + `sequence` API. Each greeting is a Framer Motion sequence step. Clean in React context; verbose setup.

**Pros:** native React integration, auto `prefers-reduced-motion` via `<MotionConfig reducedMotion="user">`.
**Cons:** ~50KB bundle cost for logo-only animation. Overkill unless project already uses Framer.

**Recommendation:** WAAPI for production (same as this mockup). No dependencies, precise control, handles reduced-motion via `window.matchMedia` check before starting.

## Accessibility

- `prefers-reduced-motion: reduce` → animation not started (JS gate before `runGreetingLoop` call). Warm orb stays statically centered.
- Favicon sizes (16px, 32px) → always static. Wave gesture at those scales is sub-pixel movement with zero semantic value.
- `aria-label="tada!ify logo"` + `role="img"` on every animated SVG.
- Animation loop is infinite but uses `requestAnimationFrame`-budget-friendly WAAPI (not `setInterval` painting). No jank from background tabs — browsers throttle WAAPI in hidden tabs.
