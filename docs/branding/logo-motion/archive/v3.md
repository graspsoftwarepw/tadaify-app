# Tadaify Logo Motion v3 — bounce-back-to-center

## Sizing contract

| Element | SVG user-space (viewBox 0 0 100 100) | At 240px render |
|---|---|---|
| Large orb radius | r = 40 | 96px |
| Large orb diameter | 80 units | 192px |
| **Warm orb radius** | **r = 20 (= large orb r ÷ 2)** | **48px** |
| Warm orb diameter | 40 units | **96px = large orb radius in px** |
| Large orb canvas margin | 10 units each side | 24px |
| Warm orb travel distance (center → edge, contained) | 20 units | 48px |
| Warm orb overshoot distance (peak) | 26 units | 62px |

The sizing rule (locked in `brand-lock.md §4`): **warm orb diameter = large orb radius**. In user-space this means r=20 for the warm orb when the large orb is r=40. At any rendered size, the warm orb is exactly ½ the diameter of the large orb — large enough to read as a co-equal personality, not a satellite detail.

When the warm orb center travels to the edge position (20 units from center), its outer edge is flush with the large orb's outer edge (large orb outer edge at 40 units, warm orb outer edge at 20 + 20 = 40 units). The warm orb is fully contained, just touching the perimeter.

During the overshoot peak (26 units from center), the warm orb's outer edge clips ~6 units outside the large orb. This is intentional — the brief spill is what creates the spring "pop" feeling on the out-trip.

**Greeting sequence: N → E → S → W** (cardinal compass order, clockwise from top).
Rationale: each direction is distinctly different (top, right, bottom, left) and the clockwise
progression has implicit directionality without feeling like a spinner (because the full
return-to-center between each greeting breaks any orbital continuity).

**Translate values per direction** (SVG user-space from center 50,50):

| Direction | Settled position | Overshoot position |
|---|---|---|
| North | translate(0, -20) | translate(0, -26) |
| East | translate(+20, 0) | translate(+26, 0) |
| South | translate(0, +20) | translate(0, +26) |
| West | translate(-20, 0) | translate(-26, 0) |

## Why this change — v2 failure analysis

**v2 failure 1 — warm orb too small.** In v2 the warm orb was drawn at r=8 in a 100-unit viewBox where the large orb is r=40. That is a 5:1 diameter ratio — the warm orb appeared as a small, secondary detail rather than as a presence with equal personality to the large orb. At 240px the warm orb was only ~38px wide, easily mistaken for a highlight or inner shadow rather than a distinct character. The brand intention ("two orbs of relative equality, one the stage, one the performer") was lost entirely at any size below 120px. The fix is to enforce the sizing contract from brand-lock.md: warm orb diameter = large orb radius, making the warm orb visually substantial at every scale.

**v2 failure 2 — arc-sweep pattern reads as spinner.** v2's motion was a continuous sweep: center → GP1 → GP2 → GP3 → GP4 → return-arc → center, never returning to center between greeting points. The warm orb thus orbited in a roughly elliptical path, which reads identically to a loading indicator or a planet in orbit. There was no moment of "re-grounding" at center between greetings — no sense that each direction was a separate, intentional acknowledgement. The bounce-back-to-center pattern in v3 forces a complete return home between each greeting, making the center state (the static brand mark) visible 4× per cycle and establishing the center as the "identity anchor" the warm orb always returns to.

## Accessibility

`@media (prefers-reduced-motion: reduce)` collapses the animation to a static state:

```css
@media (prefers-reduced-motion: reduce) {
  .orb-warm-animated {
    animation: none;
    transform: translate(0px, 0px); /* warm orb centered — static lock state */
  }
}
```

The result is the standard static brand mark (warm orb centered on large orb, per brand-lock.md §5) — fully legible and intentional, not a broken/frozen mid-animation frame.

Favicon sizes (16px, 32px) always render static regardless of user preference, because the warm orb's 6-unit travel is ~1px at 16px — invisible and potentially jittery. Animation is suppressed by omitting the `.orb-warm-animated` class on those SVG instances.

## Implementation hint — React component

When implementing as a React component, choose between:

### Option A — CSS `@keyframes` (recommended for production)

**Tradeoffs:**
- Pro: zero JS bundle cost, GPU-composited (transform only), respects `prefers-reduced-motion` natively via media query, no flash-of-unstyled-animation on hydration.
- Pro: the spring-overshoot approximation (3-keyframe out-trip with intermediate positions) is entirely achievable in CSS.
- Con: timing adjustments require counting percentages by hand; no physics-based spring — the overshoot is a static approximation.
- Con: difficult to make timing dynamic (e.g. speed up on hover, pause on focus).

Recommended for the nav logo and hero logo where motion is ambient / decorative and should be lightweight.

### Option B — Framer Motion `useAnimate` + spring

**Tradeoffs:**
- Pro: true spring physics (`stiffness`, `damping`, `mass`) — the overshoot becomes natural rather than approximated; the settle curve adapts to any interruption.
- Pro: trivial to pause on `aria-hidden`, speed-scale, or chain with page transitions.
- Con: adds ~40KB to the bundle (mitigated if Framer Motion is already in the stack for other animations).
- Con: SSR hydration requires care (`useEffect` or `m.circle` with `initial` prop to avoid layout shift).

Recommended if the logo is used inside an animated onboarding sequence or interactive landing page where it reacts to user scroll/hover.

**Recommendation:** ship CSS `@keyframes` for v1 of the component. Upgrade to Framer Motion only if the product needs the logo to respond dynamically to user interaction (e.g. the warm orb peeks toward the user's cursor position — a plausible later enhancement).
